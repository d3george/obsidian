## 引言：当代理与 SSH “打架”时

对于经常在 macOS 上使用 ClashX 等代理工具，同时又需要 SSH 连接到远程服务器的开发者或用户来说，可能会遇到一个令人困惑的场景：代理工作正常，网页浏览顺畅，但只要一开启 **“规则模式”**，尝试 SSH 连接到服务器时就会立即报错：

> `Connection closed by 142.171.179.106 port 22`

这行冰冷的文字让你摸不着头脑：我的服务器明明开着，为什么连接会被\*\*“关闭”\*\*？别担心，这不是你服务器的问题，而是你的代理工具“管得太宽”了。

## 🔍 错误解析：谁在关闭你的连接？

我们先来分析这行报错的真正含义：

  * **`142.171.179.106`：** 你尝试连接的目标服务器 IP 地址。
  * **`port 22`：** SSH 协议的标准端口。
  * **`Connection closed by...`：** **重点在这里！** 这个消息表明 TCP 连接已经成功建立，但远程主机（或中间的某个设备）在 SSH 握手开始前或刚开始时，主动发出了终止连接的信号。它接受了你的请求，但立即把你“请”了出去。

### 🚨 核心原因：端口 22 被代理拦截

当你开启 ClashX 的 **“规则模式”** 时，你的所有网络流量都会根据配置文件中的规则进行匹配和转发。

1.  **规则匹配的陷阱：** 大多数用户的配置中，访问国外的 IP 或未被明确标注的流量，最终都会被一个类似 `MATCH, PROXY` 的规则捕获，从而导向你选择的代理节点。
2.  **代理节点的策略：** **出于安全考虑，绝大多数商业代理服务（机场）的节点都会在网关层面屏蔽敏感端口，尤其是：**
      * **端口 22 (SSH)：** 防止用户利用节点进行服务器攻击、扫描或爆破。
      * **端口 25/465 (SMTP)：** 防止用户发送垃圾邮件。
3.  **连接被切断：** 当你的 SSH 流量（端口 22）被 ClashX 转发到代理节点时，代理节点会识别出这是一个被禁止的端口请求，并立即切断连接，你的客户端于是收到了 `Connection closed` 的错误。

-----

## ✅ 终极解决：强制 SSH 流量走直连 (DIRECT)

解决这个问题的思路只有一个：告诉 ClashX，所有发往 **22 端口**的流量，都必须 **“直连”（DIRECT）**，绕过代理节点。

### 方法一：修改 Clash 配置文件（推荐）

这是最一劳永逸的方法，因为它直接写入了代理逻辑。

1.  **打开配置文件夹：**

      * 点击 ClashX 菜单栏图标，选择 **配置 (Config)** -\> **打开本地配置文件 (Open Config Folder)**。
      * 找到你正在使用的 `.yaml` 配置文件，用文本编辑器打开。

2.  **添加直连规则：**
    找到文件中的 `rules:` 部分。你需要将 SSH 直连规则添加到**列表的顶部**，因为 Clash 的规则是**从上到下匹配**，一旦匹配成功就会停止。

    **最佳方案：针对所有 22 端口直连**
    在 `rules:` 列表的最上方（但要低于像 `DOMAIN-KEYWORD` 或 `IP-CIDR` 这样的自定义规则）添加以下一行：

    ```yaml
    rules:
      # --- 新添加的规则必须靠前 ---
      - DST-PORT,22,DIRECT 
      # --------------------------
      - GEOIP,CN,DIRECT
      - MATCH,PROXY # 这是你的兜底规则
    ```

      * `DST-PORT,22,DIRECT`：表示所有目标端口为 22 的流量都走直连。

    **可选方案：针对特定 IP 直连**
    如果你只需要连接固定的服务器 IP，可以更精确地添加规则（将示例 IP 替换成你的服务器 IP）：

    ```yaml
    rules:
      - IP-CIDR,142.171.179.106/32,DIRECT
      # ...
    ```

3.  **重载配置：**
    保存 `.yaml` 文件后，回到 ClashX 菜单，点击 **“重载配置” (Reload Config)**。

现在，你再次尝试 SSH 连接，流量将不再经过代理，连接应该可以立即成功。

### 方法二：检查并清除终端环境变量（排查干扰项）

如果你在终端（如 iTerm2 或 Terminal）中手动设置了代理环境变量，它可能会覆盖 ClashX 的系统设置。

1.  **检查环境变量：**
    在你的 shell 中（如 Bash 或 Zsh）输入：
    ```bash
    echo $http_proxy
    echo $all_proxy
    ```
2.  **清除环境变量：**
    如果它们有输出（例如 `http://127.0.0.1:7890`），请执行以下命令来临时清除它们：
    ```bash
    unset http_proxy https_proxy all_proxy
    ```
    **注意：** 这只在当前的终端会话中有效。如果需要永久清除，请修改你的 `.bashrc` 或 `.zshrc` 文件。

## 总结

SSH 连接失败并报 `Connection closed`，在 ClashX 规则模式下几乎可以肯定是**代理规则过于宽泛**，导致将不该代理的敏感协议流量导向了受限的代理节点。

牢记这个解决方案：**将目标端口 22 的流量配置为 `DIRECT` 直连，是保障 SSH 稳定连接的关键！**
