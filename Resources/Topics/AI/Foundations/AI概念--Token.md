# AI Token

## 定义

**一句话：** Token 是 AI 模型处理文本的**「原子计量单位」**——既不是字符也不是单词，而是模型训练时学到的、语义相对完整的最小文本片段，同时也是**计费和上下文限制的基本单元**。

**比喻：** 像**乐高积木的「标准小块」**：一篇文章不是按「字母」或「整句」卖的，而是按用了多少块乐高来计价。同样的概念，中英文「块」的大小不同（中文一个标点可能占 3 块，英文一个单词可能 1-2 块），但收银台只认「总块数」。

> **对应关系**：「乐高小块」对应 **Token**；「一篇文章」对应 **输入 + 生成的总文本**；「块数」对应 **Token 计数**；「收银台」对应 **模型计费/配额系统**；「中英文块大小不同」对应 **字符编码差异**（如 UTF-8 字节数）。不较真之处：真实分词规则复杂（如 BPE、SentencePiece），且不同模型「块」的大小标准不同，但「按块计价、有上限」的直觉成立。

## 核心要点

- **本质**：模型训练和推理时的**基本处理单元**，是整数 ID（如 15496 → "Hello"）
- **与字符/单词的区别**：
  - 英文："ChatGPT" 可能是 2 个 token（Chat + GPT）；标点符号通常是独立 token
  - 中文："你好世界" 可能是 4-6 个 token；一个汉字通常比英文字母消耗更多 token
  - 数字、代码、特殊符号也有各自的切分规则
- **为何重要**：
  - **上下文窗口限制**：模型一次能处理的 token 总数有上限（如 4K/8K/128K/200K）
  - **计费依据**：几乎所有 AI API（OpenAI、Anthropic、Kimi 等）都按 **输入 token + 输出 token** 计费
  - **性能指标**：上下文越长，推理耗时/成本通常越高
- **估算工具**：各大模型提供商都有 tokenizer 可视化工具（如 OpenAI Tokenizer、Tiktoken）

## 与现有概念的关系

| 概念 | Token 的作用 |
|------|-------------|
| [[AI概念--上下文]] | 上下文窗口的「容量」用 token 计量；`@` 挂载的文件、对话历史、系统提示都消耗 token 配额 |
| [[AI概念--幻觉]] | 上下文 token 超限被截断时，模型可能「遗忘」前文约束，增加幻觉风险 |
| [[AI概念--Skill]] | Skill 按需加载机制的目的之一就是**省 token**——不重复贴长 prompt |

| [[AI概念--Agent]] | Agent 多轮执行时，累计的观察-思考-行动链持续消耗 token；长任务需规划压缩或分段 |

## 应用场景

- **预算控制**：设计 Prompt 时估算 token，避免超出预算或上下文上限
- **上下文管理**：Skill 的「按需加载」、RAG 的「摘要后塞入」都是为了**在 token 限制内装最多信息**
- **分块策略**：RAG 把文档切成「适合上下文的小块」，块大小用 token 衡量
- **模型选型**：短任务用 4K 窗口的便宜模型；长文档分析用 128K/200K 的模型

## 一句话总结

**Token 是 AI 世界的「硬通货」**——它决定你能「记住」多少、能「生成」多少、要「付多少费」。理解 token，才能理解为什么长文档要摘要、为什么 Skill 要按需加载、为什么上下文会「溢出」。
