[[📺-How Git Works Explained in 4 Minutes]]

很多初学者认为代码只存在于两个地方：**本地（Local）**和**远程（Remote）**。但 ByteByteGo 指出，要真正理解 Git，我们需要意识到代码其实是在**四个不同的位置**流转。理解了这四个区域，你就能明白每条 Git 指令到底把代码搬运到了哪里。

## 剖析四大区域
我们要先搞清楚这四个区域的具体定义，特别是**暂存区**这个初学者容易忽视的概念。
### 💻 工作目录 (Working Directory)

>这是你的私人工作台，你在这里进行所有的编辑和创造。

![](./assets/How%20Git%20Works/image-2025-11-23-20-38-32.png)

- **文件状态**：
    
    - **未追踪 (untracked)**：新创建的文件，Git 还不知道它的存在。
        
    - **已修改 (modified)**：Git 知道这个文件，但你对它做了修改，这些修改还没被记录。
        
- **关键动作**：`git add`。这是离开工作目录的唯一出口。它告诉 Git：“嘿，把这几个文件的当前状态记录下来，准备下一步操作。”
### 📦 暂存区 (Staging Area / Index)

> 这是 Git 最独特的设计之一，一个提交前的“缓冲区”或“准备室”。

![](./assets/How%20Git%20Works/image-2025-11-23-20-39-57.png)

- **核心概念**：**快照 (Snapshot)**。暂存区里存放的，是你希望包含在下一次提交中的文件状态的快照。你可以把它想象成在拍照前摆好的姿势。
    
- **关键动作**：
    
    - 流入：`git add` 将文件从工作目录带到这里。
        
    - 流出：`git commit` 将这里的快照永久保存起来
### 🗄️ 本地仓库 (Local Repository)

> 这是你的私人档案馆，所有提交的历史记录都安全地保存在这里。

![](./assets/How%20Git%20Works/image-2025-11-23-20-41-04.png)

- **核心功能**：**版本控制**。这里存储着项目完整的提交历史（如图中的 `c1`, `c2`, `c3`）。每一个圆点都是一个永久的记录，你可以随时回溯到任何一个版本。
    
- **关键动作**：
    
    - 流入：`git commit` 将暂存区的快照保存为一个新的提交。
        
    - 流出：`git push` 将本地的新提交发送到远程服务器。
### ☁️ 远程仓库 (Remote Repository)

> 这是团队协作和代码备份的云端中心，比如 GitHub。

![](./assets/How%20Git%20Works/image-2025-11-23-20-41-45.png)

- **核心功能**：**协作与共享**。图中的小人代表了你的团队成员，大家都在与这个中心仓库交互。
    
- **关键动作**：
    
    - 流入：`git push` 将你的成果分享给团队。
        
    - 流出：`git pull`。这是一个组合技，它实际上是先 `git fetch`（把远程的更新抓取到本地，但不合并），然后 `git merge`（把抓取下来的更新合并到你的当前分支）。


## ➡️ 追踪代码流转

我们模拟一次代码提交的过程，看看 `git add`、`commit` 和 `push` 是如何一步步驱动代码移动的。

![](./assets/How%20Git%20Works/image-2025-11-23-20-43-14.png)


### 拉取代码
这张图清晰地展示了 Git 中从远程仓库获取更新并同步到本地的两种主要方式：`git pull` 的直接同步，以及 `git fetch` 与 `git merge` 的分步操作。

![](./assets/How%20Git%20Works/image-2025-11-23-20-50-48.png)

让我们逐一解释图中箭头的含义：

1. **`git pull` (蓝色实线箭头)**：
    
    - **路径**：从**远程仓库**直接指向你的**工作目录**。
        
    - **含义**：`git pull` 是一个组合命令，它会执行以下两个步骤：
        
        1. 首先，它会从远程仓库**抓取 (fetch)**所有最新的更改到你的本地仓库（但这些更改不会立即影响你的工作目录）。
            
        2. 然后，它会立即将这些抓取到的远程更改**合并 (merge)**到你当前所在的分支，并更新你的工作目录中的文件。
            
    - **特点**：一步到位，将远程更新完全同步到你的本地分支和工作目录。如果存在冲突，会直接进入冲突解决状态。
        
2. **`git fetch` (红色实线箭头)**：
    
    - **路径**：从**远程仓库**指向你的**本地仓库**。
        
    - **含义**：`git fetch` 只负责从远程仓库**下载**最新的所有分支和提交历史到你的本地仓库。
        
    - **特点**：它只会更新你的本地仓库中关于远程分支的引用（例如 `origin/main`），但**不会**修改你当前的工作目录中的文件，也不会自动合并到你当前的分支。这给你了一个在合并前查看远程更新、决定何时合并的“预览”机会。
        
3. **`git merge` (红色虚线箭头)**：
    
    - **路径**：从**本地仓库**指向你的**工作目录**（更准确地说，是从本地仓库中的一个分支合并到当前工作目录所对应的本地分支）。
        
    - **含义**：当你执行 `git fetch` 后，远程的最新代码已经存在于你的本地仓库中（以远程分支的形式）。此时，如果你决定要将这些远程更改集成到你正在工作的本地分支，就需要执行 `git merge` 命令。例如，`git merge origin/main` 会将本地仓库中 `origin/main` 所指向的远程最新状态合并到你当前的工作分支，并同步到你的工作目录。
        
    - **特点**：将两个或多个分支的更改合并在一起，生成一个新的合并提交（除非是快进合并）。这个操作会直接影响你当前工作目录中的文件。

### Git 分支集成策略详解：`merge` vs `rebase` vs `squash commit`

这三种策略都是将一个分支（通常是特性分支 `feature`）的更改集成到另一个分支（通常是主分支 `main`）的方法，但它们处理提交历史的方式截然不同。


#### `git merge` (合并)

**核心思想：** 保留所有分支的原始提交历史，并在集成时创建一个新的“合并提交”来连接两个分支的演变。
![](./assets/How%20Git%20Works/image-2025-11-23-21-07-29.png)

**工作原理：** 当你在 `main` 分支上执行 `git merge feature` 时：

1. Git 会找到 `main` 分支和 `feature` 分支最近的共同祖先 `C1`。
    
2. 它会比较 `feature` 分支和 `main` 分支各自从共同祖先以来的所有更改。
    
3. 如果存在冲突，你需要手动解决。
    
4. 如果没有冲突或冲突解决后，Git 会创建一个**新的合并提交 (Merge Commit)**。这个提交有**两个父提交**：一个是 `main` 分支合并前的最新提交，另一个是 `feature` 分支的最新提交`F`。
    
5. `main` 分支的 HEAD 指针会移动到这个新的合并提交。
    

**提交历史特点：**

- **网状结构**：提交历史会呈现出分支合并时的“岔路口”和“汇合点”，形成一个非线性的网状结构。
    
- **完整保留**：`feature` 分支上的所有原始提交（A, B, C 等）都会完整地保留在历史中。
    
- **非破坏性**：不会改变任何已有提交的哈希值或内容，是最安全的集成方式。
    

**何时使用：**

- 当你希望保留项目完整的、真实的分支开发历史时。
    
- 在合并到长期存在的分支（如 `main`/`master` 或 `develop`）时，通常是安全的默认选择。
    
- 当你在一个公共分支（已经被 `push` 到远程，且有其他人在其上工作）上操作时，`merge` 是唯一安全的选择，因为 `rebase` 会改写历史。

**等效命令：**
```bash
# 首先切换到目标分支 (main)
git checkout main
# 然后执行合并操作
git merge feature
```

#### `git rebase` (变基)

**核心思想：** 重写一个分支的提交历史，使其看起来像是在另一个分支的最新提交之后“重新播放”一遍，从而创建一条干净、线性的历史。
![](./assets/How%20Git%20Works/image-2025-11-23-21-10-38.png)

**工作原理：** 当你在 `feature` 分支上执行 `git rebase main` 时：

1. Git 会找到 `feature` 分支和 `main` 分支的最近共同祖先 `C1`。
    
2. 它会把你 `feature` 分支上，从共同祖先`C1`以来的所有提交（例如 A, B, C）**临时存起来**。
    
3. 然后，它会将 `feature` 分支的基点（base）移动到 `main` 分支的最新提交上`E`。
    
4. 接着，它会把你之前临时存起来的那些提交（A, B, C）**逐个应用 (re-apply)** 到新的基点`E`之后。在这个过程中，这些提交的哈希值会发生改变，因为它们的父提交变了。
    
5. 如果应用某个提交时发生冲突，你需要解决冲突，然后用 `git rebase --continue` 继续。
    
6. 完成 `rebase` 后，`feature` 分支的历史看起来就像是直接从 `main` 分支的最新提交开始创建的。
    

**提交历史特点：**

- **线性结构**：历史记录非常干净，像一条直线，没有合并提交，非常容易阅读。
    
- **提交哈希值改变**：`rebase` 过程中，“重新应用”的提交会生成新的哈希值，从 Git 的角度来看，它们是全新的提交。
    
- **改写历史**：因为它改变了提交的哈希值，所以 `rebase` 是一种**破坏性操作**。
    

**何时使用：**

- 在你将本地特性分支的工作集成到最新的 `main` 分支之前，以保持 `main` 分支历史的清洁和线性。
    
- **切勿对已经推送到远程的公共分支进行 `rebase`！** 因为它会改写历史，导致其他基于旧历史的团队成员无法正常工作，从而引发冲突和混乱。
    
- 清理本地分支的零碎提交（通常结合交互式 `rebase` `git rebase -i`）。

**等效命令：**
```bash
# 首先切换到要变基的分支 (feature)
git checkout feature
# 然后执行变基操作，目标是 main 分支的最新状态
git rebase main
# 变基完成后，再切换回 main 分支并执行快进合并
git checkout main
git merge feature # 这通常是一个快进合并 (Fast-Forward Merge)，因为 feature 现在直接是 main 的后继
```

#### `Squash Commit` (压缩提交)

**核心思想：** 将一个分支上的多个提交压缩成一个单一的、有意义的提交。这通常与 `rebase` 或 `merge` 结合使用。
![](./assets/How%20Git%20Works/image-2025-11-23-21-16-15.png)

**工作原理：** `Squash Commit` 本身不是一个独立的 Git 命令，而是一种通过 `git rebase -i` (交互式变基) 或 `git merge --squash` 实现的策略。

**通过 `git rebase -i` 实现：**

1. 当你执行 `git rebase -i HEAD~N` (N 是你想压缩的提交数量) 时，Git 会打开一个交互式编辑器。
    
2. 在编辑器中，你可以选择将某些提交标记为 `squash` (或 `fixup`)，将它们与前面的提交合并。
    
3. 你会被要求为这些合并后的提交编写一个新的、简洁的提交信息。
    
4. 结果是，你特性分支上的多个零碎提交（例如，"WIP: add part 1", "fix bug", "add part 2"）会被整合成一个高质量的提交（例如 "feat: implement user authentication"）。
    

**通过 `git merge --squash` 实现：**

1. 在 `main` 分支上执行 `git merge --squash feature`。
    
2. 这会将 `feature` 分支的所有更改应用到你的 `main` 分支的**工作目录和暂存区**，但**不会创建合并提交**，也不会保留 `feature` 分支的原始提交历史。
    
3. 你需要手动执行 `git commit`，为这些被压缩的更改编写一个提交信息。
    

**提交历史特点：**

- **极致简洁**：特性分支上的所有工作被浓缩成一个或少数几个提交，使得 `main` 分支的历史非常干净和精炼。
    
- **丢失中间历史**：特性分支上的原始原子提交历史会被丢弃（如果你不保存 `feature` 分支的话）。这意味着你无法再追溯到 `feature` 分支上每个小步骤的提交。
    
- **改写历史**：与 `rebase` 类似，`squash` 也会创建新的提交哈希值，因此也是一种破坏性操作。
    

**何时使用：**

- 当你的特性分支上有很多零碎的、过程性的提交（例如，"WIP", "fix typo", "test commit"），而你希望在合并到 `main` 分支时只保留一个有意义的、描述性的提交。
    
- 当你希望 `main` 分支的历史是高层次的，每个提交都代表一个完整的功能或大的修复。
    
- 在 PR (Pull Request) 或 Code Review 流程中，通常要求将一个功能的所有相关更改压缩成一个有意义的提交再合并。

**等效命令(通过交互式 rebase 实现)：**

```bash
# 假设你在 feature 分支，想压缩最近 3 个提交
git rebase -i HEAD~3
# 在打开的编辑器中，将除了第一个提交之外的其余提交前的 'pick' 改为 'squash' (或 's')
# 保存并关闭编辑器，然后编写新的提交信息
```

**等效命令(通过 merge --squash 实现)：**

```bash
git checkout main
git merge --squash feature
git commit -m "feat: implement user authentication"
```


#### 三者比对

| 特性         | `git merge`                 | `git rebase`                     | `squash commit`                  |
| :----------- | :-------------------------- | :------------------------------- | :------------------------------- |
| **历史结构** | 网状，保留完整分支历史      | 线性，重写特性分支历史           | 线性，特性分支多个提交压缩为单个 |
| **合并提交** | 会创建新的合并提交          | 不会创建合并提交                 | 不会创建合并提交 (但会创建新提交) |
| **原始提交** | 完整保留                    | 原始提交哈希值改变，是“新”提交 | 原始小提交被丢弃/合并            |
| **破坏性** | 否 (非破坏性)               | 是 (改写历史)                    | 是 (改写历史)                    |
| **适用场景** | 保留真实开发历史，公共分支  | 追求干净线性历史，个人特性分支   | 整理零碎提交，保持主分支历史简洁 |
